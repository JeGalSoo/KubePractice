Trouble Shooting

핵심 트러블슈팅

#01
🔥 HIGH IMPACT
AOP 성능 로깅 ↔ ReadOnly 트랜잭션 충돌
증상

게시판 목록 조회 시 500 Internal Server Error 발생, 강제 로그인 화면 리다이렉트

원인

@Transactional(readOnly=true) 환경에서 PerformanceAspect가 DB에 save() 시도 → Hibernate가 쓰기 차단

해결

TransactionTemplate + PROPAGATION_REQUIRES_NEW로 성능 로그 저장을 완전히 격리된 새 트랜잭션으로 분리

// Before: 메인 트랜잭션과 충돌
performanceLogRepository.save(log);  // ❌ readOnly 트랜잭션에서 쓰기
// After: 독립된 격리 트랜잭션
transactionTemplate.setPropagationBehavior(PROPAGATION_REQUIRES_NEW);
transactionTemplate.execute(status -> { performanceLogRepository.save(log); return null; }); // ✅
#02
⚡ PERFORMANCE
HikariCP 데드락 (K8s 부하 테스트)
증상

K6 500명 동시 요청 시 Connection is not available, request timed out after 30000ms 대량 발생, 에러율 100%

원인

Tomcat 스레드 200개가 각각 커넥션 점유. REQUIRES_NEW가 같은 스레드에서 2번째 커넥션 요구 → 교착상태(Deadlock)

해결

Tomcat 최대 스레드 50개 제한 + HikariCP 풀 150으로 확장 + MySQL max_connections=2000 상향. 스레드 수 < 커넥션 수 유지로 데드락 원천 차단

# ConfigMap (backend-dev.yml) 수정
server.tomcat.threads.max=50                          # 스레드 제한 (200→50)
spring.datasource.hikari.maximum-pool-size=150        # 풀 확장
#03
🌐 MSA
Kafka Advertised Listener K8s 통신 불가
증상

K6 부하 테스트 p95 응답시간 18.5초, 에러율 100%. 백엔드 로그에 Kafka 연결 타임아웃 무한 반복

원인

Kafka가 ADVERTISED_LISTENERS로 localhost:9092 반환 → K8s 파드 내부에서 자기 자신을 찌르며 60초 타임아웃 무한 대기

해결

KAFKA_ADVERTISED_LISTENERS를 host.docker.internal:9092로 수정하여 파드가 호스트 머신의 Kafka에 정확히 도달하도록 통신 경로 수정

# docker-compose.yml
Before: KAFKA_ADVERTISED_LISTENERS: PLAINTEXT://kafka:29092,PLAINTEXT_HOST://localhost:9092  ❌
After:  KAFKA_ADVERTISED_LISTENERS: PLAINTEXT://kafka:29092,PLAINTEXT_HOST://host.docker.internal:9092  ✅















#04
🔄 K8s
HPA 스케일아웃 중 EOF 에러
증상

트래픽 급증 시 211초 부근에서 1.85% EOF 에러 발생

원인

K8s가 컨테이너 시작 즉시 파드 준비 완료 판단 → Spring Boot 로딩 중 트래픽 전달 → 소켓 거부(EOF)

해결

readinessProbe + livenessProbe 추가. TCP 포트 8080 실제 열릴 때까지 트래픽 대기 → 에러율 0%

readinessProbe:
  tcpSocket:
    port: 8080
  initialDelaySeconds: 20   # Spring Boot 기동 대기
  periodSeconds: 5
#05
🗄️ ES
Elasticsearch Race Condition
증상

파드 1→5개 스케일아웃 시 15~19초 구간에 다수 EOF, 파드 4개 RESTARTS 증가

원인

4개 파드 동시에 ES boards 인덱스 자동 생성 시도 → 1개만 성공, 나머지 3개 resource_already_exists_exception으로 Crash

해결

K8s Self-Healing으로 파드 자동 재시작. 2차 기동 시 인덱스 존재 → 정상 구동. K8s 자가 치유 메커니즘 실체험

#06
🔗 Bean
연쇄 Bean 생성 실패 (3단계 순차 해결)
증상

UnsatisfiedDependencyException 연쇄 발생으로 Spring Boot 기동 완전 실패

원인

① Flyway 이력 없음 → JPA 빈 실패
② ES @Id 잘못 임포트 → ES Repository 실패
③ OAuth2 설정 누락 → ClientRegistration 실패

해결

스택 트레이스 역추적으로 근본 원인 순차 식별: ① baseline-on-migrate: true → ② springframework.data.annotation.Id로 수정 → ③ OAuth2 placeholder 추가
