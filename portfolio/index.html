About Project

프로젝트 개요

단일 모놀리식 서버의 한계를 극복하고, 실제 대규모 서비스를 운영하는 방식으로 설계된 MSA 기반 웹 애플리케이션입니다.

MSA 아키텍처

5개의 독립적인 마이크로서비스로 구성하여 각 서비스의 독립적 배포 및 스케일링을 구현했습니다.

실시간 채팅

WebFlux + WebSocket + Kafka 조합으로 논블로킹 고성능 실시간 채팅 시스템을 독립 서버로 분리 운용합니다.

고성능 검색

Elasticsearch + Nori 한글 형태소 분석기로 LIKE 쿼리 대비 압도적인 풀텍스트 검색 성능을 달성했습니다.

성능 모니터링

AOP 기반 커스텀 성능 로깅과 Prometheus + Grafana로 시스템 전 구간 실시간 모니터링을 구축했습니다.

Kubernetes 배포

K8s HPA 오토스케일링으로 트래픽 폭증 시 자동 확장되는 무중단 운영 환경을 구축하고 부하테스트로 검증했습니다.

Trouble Shooting

핵심 트러블슈팅

단순한 기능 구현을 넘어, 실제 운영에서 마주치는 복잡한 문제들을 직접 진단하고 해결했습니다.

🔥 HIGH IMPACT
#01

AOP 성능 로깅 ↔ ReadOnly 트랜잭션 충돌

증상

로그인 후 게시판 목록 조회 시 500 Internal Server Error 발생 및 강제 로그인 화면 리다이렉트

원인

@Transactional(readOnly = true) 환경에서 PerformanceAspect가 성능 로그를 DB에 save() 시도 → Hibernate가 쓰기 작업 차단하며 예외 발생

해결

TransactionTemplate 도입, PROPAGATION_REQUIRES_NEW로 성능 로그 저장 로직을 완전히 격리된 새 트랜잭션으로 분리. 파라미터 250자 Truncate 방어 로직 추가

// 기존: 메인 트랜잭션과 충돌
performanceLogRepository.save(log);  // ❌ readOnly 트랜잭션에서 쓰기 시도

// 개선: 새로운 격리 트랜잭션
transactionTemplate.setPropagationBehavior(PROPAGATION_REQUIRES_NEW);
transactionTemplate.execute(status -> {
    performanceLogRepository.save(log);  // ✅ 독립된 트랜잭션
    return null;
});
⚡ PERFORMANCE
#02

HikariCP 데드락 (K8s 부하 테스트)

증상

K6 500명 동시 요청 시 HikariPool-1 - Connection is not available, request timed out after 30000ms 대량 발생, 에러율 100%

원인

Tomcat 스레드 200개가 각각 커넥션 1개씩 점유. PerformanceAspect의 REQUIRES_NEW가 같은 스레드에서 2번째 커넥션을 요구 → 전 스레드가 서로 커넥션을 기다리는 교착상태(Deadlock)

해결

Tomcat 최대 스레드 50개로 제한 + HikariCP 풀 사이즈 150으로 확장 + MySQL max_connections=2000 상향. 스레드 수 < 커넥션 수를 유지하여 데드락 원천 차단

# ConfigMap (backend-dev.yml) 수정
server.tomcat.threads.max=50           # 스레드 제한 (200 → 50)
spring.datasource.hikari.maximum-pool-size=150  # 풀 확장
# MySQL: --max_connections=2000        # 스케일아웃 대비
🌐 MSA
#03

Kafka Advertised Listener K8s 통신 불가

증상

K6 부하 테스트 p95 응답시간 18.5초, 에러율 100%. 백엔드 로그에 Kafka 연결 타임아웃 무한 반복

원인

Kafka가 ADVERTISED_LISTENERS로 localhost:9092 반환 → K8s 파드 내부에서 자기 자신(파드 내부)을 찌르며 60초 타임아웃 무한 대기

해결

docker-compose.yml에서 KAFKA_ADVERTISED_LISTENERS를 host.docker.internal:9092로 수정하여 파드가 호스트 머신의 Kafka에 정확히 도달하도록 통신 경로 수정

# docker-compose.yml - 수정 전
KAFKA_ADVERTISED_LISTENERS: PLAINTEXT://kafka:29092,PLAINTEXT_HOST://localhost:9092  # ❌

# 수정 후
KAFKA_ADVERTISED_LISTENERS: PLAINTEXT://kafka:29092,PLAINTEXT_HOST://host.docker.internal:9092  # ✅
🔄 K8s
#04

HPA 스케일아웃 중 EOF 에러 (Readiness Probe 부재)

증상

트래픽 200명→500명 급증 시 211초 부근에서 1.85%의 EOF 에러 발생

원인

K8s가 컨테이너 프로세스 시작 즉시 파드 준비 완료로 판단 → Spring Boot가 아직 10~20초 로딩 중인데 게이트웨이가 트래픽을 전달 → 소켓 연결 거부(EOF)

해결

Deployment YAML에 readinessProbe + livenessProbe 추가. K8s가 TCP 포트 8080이 실제 열릴 때까지 기다렸다가 트래픽을 전달하도록 설정 → 1.85% 에러 완전 근절

readinessProbe:
  tcpSocket:
    port: 8080
  initialDelaySeconds: 20   # Spring Boot 기동 대기
  periodSeconds: 5
livenessProbe:
  tcpSocket:
    port: 8080
  initialDelaySeconds: 30
🗄️ ES
#05

Elasticsearch Race Condition (HPA 스케일아웃)

증상

파드 1→5개 스케일아웃 시 15~19초 구간에 다수의 EOF 에러, 파드 4개의 RESTARTS 카운트 증가

원인

4개의 파드가 동일 밀리초에 ES boards 인덱스 자동 생성 시도 → 1개만 성공, 나머지 3개는 resource_already_exists_exception 예외로 Crash

해결

K8s Self-Healing 기능으로 파드 자동 재시작. 두 번째 기동 시 인덱스 이미 존재 → 충돌 없이 정상 구동. 이 과정을 통해 K8s 자가 치유 메커니즘을 실제로 체험하고 이해

$ kubectl get pod -l app=backend \
  -o custom-columns=NAME:.metadata.name,RESTARTS:.status.containerStatuses[0].restartCount
NAME                       RESTARTS
backend-7c7f8554f4-7mm6d   1    # Self-Healing 후 정상화
backend-7c7f8554f4-9kplq   0
🔗 Bean
#06

연쇄 Bean 생성 실패 (3단계 순차 해결)

증상

UnsatisfiedDependencyException 연쇄 발생으로 Spring Boot 기동 완전 실패

원인

① Flyway 스키마 이력 없음 → JPA EntityManagerFactory 빈 생성 실패
② ES Document @Id를 JPA용으로 잘못 임포트 → ES Repository 빈 생성 실패
③ OAuth2 client 설정 누락 → ClientRegistrationRepository 빈 생성 실패

해결

스택 트레이스 역추적으로 근본 원인 순차 식별: ① baseline-on-migrate: true → ② org.springframework.data.annotation.Id로 임포트 수정 → ③ OAuth2 placeholder 설정 추가
